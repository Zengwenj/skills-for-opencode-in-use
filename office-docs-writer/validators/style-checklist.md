# 格式规范清单

核查文档格式是否符合 guidelines/formal-style.md 的要求：

## 基准版式说明

本文档的格式核查以以下基准为准（详见 guidelines/formal-style.md）：

- **默认**：普通页面版式（上下2.54cm，左右3.18cm，页眉1.50cm，页脚1.75cm）
- **公文版式**（用户明确要求时）：上37mm，下35mm，左28mm，右26mm

## 一、页面版式（输出说明）

- [ ] 已在文档说明或输出提示中注明适用版式
- [ ] 公文版式仅在用户明确要求时使用
- [ ] 红头仅在文种规范要求设置，或用户明确要求设置红头时启用
- [ ] 未明确要求红头时，不默认添加发文机关、发文字号或红色分隔线版头结构

## 二、文字格式

- [ ] 正文行距：28磅（固定值）
- [ ] 标题行距：40磅（固定值）
- [ ] 正文非表格段落与1-4级标题首行缩进：2字符（使用 XML w:firstLineChars="200"）
- [ ] 主标题字号：二号
- [ ] 正文字号：三号

## 二-B、DOCX 前置检查（仅当输出为 `.docx`）

- [ ] 已检查目标中文字体是否已安装
- [ ] 已确认生成链能写入 `w:eastAsia`
- [ ] 已确认生成链能写入 `w:firstLineChars`
- [ ] 已确认生成链能写入 Heading 样式或等效 `w:outlineLvl`
- [ ] 若任一能力缺失，已显式报告风险而非继续宣称“格式合规”
- [ ] 若目标字体缺失并发生替代，已明确写出风险说明，且未把替代结果表述为“完全合规”

## 二-A、红头与版头（按需启用）

- [ ] 如未明确要求红头，不检查红头版头结构
- [ ] 如明确要求红头，发文机关居中排布
- [ ] 如明确要求红头，发文机关上边缘至版心上边缘为 35mm
- [ ] 如明确要求红头，发文机关字体为 方正小标宋_GBK，字号为 72号，颜色为红色
- [ ] 如明确要求红头且有发文字号，红色分隔线位于发文字号下 4mm 处
- [ ] 如明确要求红头且无发文字号，红色分隔线位于发文机关下 4mm 处
- [ ] 如明确要求红头，红色分隔线为与版心等宽的红色直线段
- [ ] 如明确要求红头，主标题位于红色分隔线下空二行

## 三、段落间距（重要）

- [ ] **段前间距：0行**（必须显式设置为 Pt(0)，不能为 None）
- [ ] **段后间距：0行**（必须显式设置为 Pt(0)，不能为 None）

## 四、字体使用

- [ ] 主标题使用方正小标宋_GBK（不加粗）
- [ ] 正文使用方正仿宋_GBK
- [ ] 一级标题使用方正黑体_GBK（不加粗）
- [ ] 二级标题使用方正楷体_GBK（不加粗）
- [ ] 三级标题使用方正仿宋_GBK（不加粗）
- [ ] 四级标题使用方正仿宋_GBK（不加粗）
- [ ] 数字和英文使用 Times New Roman
- [ ] 中文字体已通过 `w:eastAsia` 落地，不是只设置了 `w:ascii` / `w:hAnsi`
- [ ] 若使用替代字体，已在交付说明中明确目标字体与实际字体，不将替代结果视为指定字体已落地

## 五、标点符号规范

- [ ] 引号使用**中文引号**（"" 和 ''）
- [ ] 不使用英文引号 ("" 和 '')，除非用户特殊说明

## 六、字体颜色与字形（重要）

- [ ] **所有文字颜色：纯黑色 RGB(0, 0, 0) / XML `w:val="000000"`**
- [ ] 所有 run 必须显式落为纯黑色，不得保留或继承 `themeColor`
- [ ] 使用 `Heading 1/2/3/4` 时不得继承 Word 内置 Heading 样式的蓝色主题色；必须覆盖为黑色或清除 `themeColor`
- [ ] **字形：常规**（非斜体、无下划线）
- [ ] 主标题、一级标题、二级标题、正文保持常规字形
- [ ] 不得沿用“二级标题必须加粗”的旧规则

## 七、大纲级别设置（重要）

- [ ] 一级标题（一、）设置大纲级别 1级（Heading 1）
- [ ] 二级标题（（一））设置大纲级别 2级（Heading 2）
- [ ] 三级标题（1.）设置大纲级别 3级（Heading 3）
- [ ] 四级标题（（1））设置大纲级别 4级（Heading 4）
- [ ] 正文不设置大纲级别（或设为正文/Body Text）
- [ ] 不存在“一级标题被写成正文样式”的情况
- [ ] 不存在“二级标题被写成正文样式”的情况

## 八、段落结构

- [ ] 编号层级一致（一级"一、"，二级"（一）"，三级"1."，四级"（1）"）
- [ ] 列举项编号不混用
- [ ] 文档主标题**不得设置首行缩进**
- [ ] 1-4级标题设置首行缩进2字符（`w:firstLineChars="200"`）
- [ ] 正文非表格段落设置首行缩进2字符（`w:firstLineChars="200"`）
- [ ] 表格单元格内段落**不得设置首行缩进**

## 九、内容格式

- [ ] 最终文档无 Markdown 格式符号（`#`、`**`、`-`、`*`等）
- [ ] 无多余空行
- [ ] 表格格式清晰（如有）

## 十、落款

- [ ] 落款含制发单位名称
- [ ] 落款含日期（格式：XXXX年XX月XX日）

## 十一、常见错误（必须避免）

### 1. 首行缩进（关键）
- ❌ 错误：`first_line_indent = Pt(24)` —— Word显示为"0.85cm"
- ❌ 错误：`firstLineChars = '2'` —— 值不正确（应为'200'）
- ❌ 错误：设置 `w:firstLine='480'`、`Cm(...)`、`Pt(...)` 或任何 cm/twips 近似值 —— 这不是“2字符”的合规表达
- ❌ 错误：主标题套用标题缩进规则，存在 `w:firstLineChars` 或 `w:firstLine`
- ❌ 错误：表格单元格段落套用正文首行缩进，存在 `w:firstLineChars` 或 `w:firstLine`
- ✅ 正确：使用 XML `w:firstLineChars="200"` —— Word显示为"2字符"
- ✅ 正确：主标题、表格单元格内段落不写入任何首行缩进属性
- **技术实现**：
```python
from docx import Document
from docx.oxml import OxmlElement
from docx.oxml.ns import qn
from docx.shared import Pt

doc = Document()
paragraph = doc.add_paragraph()
paragraph.paragraph_format.first_line_indent = Pt(1) # 占位，触发按正确顺序插入 w:ind
pPr = paragraph._element.get_or_add_pPr()
ind = pPr.find(qn('w:ind'))
if ind is None:
    ind = OxmlElement('w:ind')
    pPr.append(ind)
ind.set(qn('w:firstLineChars'), '200')
ind.attrib.pop(qn('w:firstLine'), None) # 移除多余的物理宽度参数
```
- **重要**：`w:firstLineChars` 的单位是 1/100 字符，所以2字符必须写为 "200"
- **重要**：`w:firstLineChars` 与 `w:firstLine` 语义不同；“首行缩进2字符”只能按字符单位校验，不能接受物理长度近似值。

### 2. 段落间距
- ❌ 错误：`space_before = None` —— 继承样式，不是0行
- ✅ 正确：`space_before = Pt(0)` —— 明确段前0行

### 3. 字体颜色
- ❌ 错误：不设置 color.rgb —— 会使用主题色
- ❌ 错误：Heading 2/3 只设置样式，不覆盖 Word 内置标题样式默认蓝色主题色
- ✅ 正确：`run.font.color.rgb = RGBColor(0, 0, 0)` —— 纯黑色
- ✅ 正确：清除或覆盖 `w:themeColor`，并确保 `w:color w:val="000000"`

### 4. 字形
- ❌ 错误：不设置 italic/underline —— 将继承样式
- ✅ 正确：显式设置 `italic = False`, `underline = False`

### 5. 标题样式映射（关键）
- ❌ 错误：`一、` 段落被写成正文样式（如 Body Text、正文、`CC_Body`）
- ❌ 错误：`（一）` 段落被写成正文样式（如 Body Text、正文、`CC_Body`）
- ✅ 正确：标题编号段落必须映射到对应 Heading 1/2/3/4 或等效 `w:outlineLvl`

### 6. XML 级验收（关键）
- ✅ 检查 `w:eastAsia` 是否与目标中文字体一致
- ✅ 检查 `w:firstLineChars="200"` 是否存在于应缩进段落
- ✅ 检查主标题段落不得存在 `w:firstLineChars` / `w:firstLine`
- ✅ 检查表格单元格段落不得存在 `w:firstLineChars` / `w:firstLine`
- ✅ 检查标题段是否带有正确 `w:pStyle` 或等效 `w:outlineLvl`

## 十二、结构性校验（带版记/带表格文档）

> 详见 guidelines/footer-imprint.md

- [ ] 表格数量：文档中 `<w:tbl>` 元素数量符合预期
- [ ] 表格维度：行数 × 列数与规定结构一致（如 2行×1列）
- [ ] 版记表格各列对齐方式正确（如左空一列、右顶格）
- [ ] 单元格边框：横线仅出现在行间（`w:top`/`w:bottom` 为单线），不得出现竖线（`w:left`/`w:right` 为 nil）
- [ ] 版记表格与正文段落顺序正确：版记表格在正文末尾、`w:sectPr` 之前
- [ ] 版记表格前后不残留多余空段落
- [ ] 版记字体以历史稿实测为准，允许独立于正文字体；无历史稿时经用户确认后可使用仿宋
- [ ] 版记字体与正文允许不同（如版记使用仿宋、正文要求仿宋时视为一致）
- [ ] 版记表格后存在独立落款段落（如"（共印X份）"）

### 常见结构性错误

- ❌ 错误：版记表格出现在正文中间而非末尾
- ❌ 错误：`w:tcBorders` 四边均有线条，与"仅横线无竖线"要求冲突
- ❌ 错误：版记表格行数或列数不匹配规定结构
- ✅ 正确：版记表格位于正文末尾、`w:sectPr` 之前，仅行间横线，列数/行数匹配
- ✅ 正确：版记字体以历史稿实测为准（详见 guidelines/footer-imprint.md），无历史稿时经用户确认后可使用仿宋
