# Article Analysis Imitation

## Overview

把这项任务当成一条完整工作流，而不是一次性提示词。

你的目标通常有四个：

1. 分析同一作者多篇文章的稳定风格
2. 区分恒定特征与主题适应特征
3. 给出可执行的仿写指导
4. 在需要时沉淀出可复用的文体模板 / 风格画像，供后续适配到 `office-docs-writer`

默认优先级：思维方式 > 结构组织 > 语言标记 > 表层措辞。

核心原则：**没有跨文本证据，不下稳定风格结论；没有原文依据，不把判断包装成事实。**

## When to Use

在下列场景调用本 skill：

- 用户给出同一作者的 3-5 篇文章，希望分析文风、格式、写法或表达习惯
- 用户希望模仿某位作者的文章风格，而不是只做摘要或改写
- 用户希望把分析结果沉淀成可复用的写作模板、风格画像或 office-doc 可选风格资产
- 用户明确要求区分哪些特征是稳定的，哪些特征会随主题变化

如果只有单篇文章，不要假装已经识别出作者的稳定风格。此时只能做单篇风格分析，并明确说明稳定特征结论不足。

## Required Inputs

开始前先确认输入是否足够：

1. 最好有同一作者 3-5 篇文本样本
2. 每篇样本最好达到 200 字以上，并保留基本段落结构
3. 样本尽量覆盖不同主题或不同写作场景
4. 输入优先使用可读文本；如果来自 PDF / 图片 / OCR，先确保文本可读且段落未严重损坏
5. 如果用户想做 office-doc 兼容模板，再额外确认目标用途，例如：工作总结、汇报材料、讲话稿、简报、评论性文章

如果输入不足，先说明限制，再给出降级方案：

- 单篇：只做单篇风格拆解，不下跨文本稳定性结论，也不声称“锁定作者长期风格”
- 两篇：可做初步比较，但要把稳定特征标成暂定，且只有两篇都出现的模式才可列入暂定稳定项
- 100-199 字样本：只做有限观察，不承担微观风格的强判断
- 少于 100 字的样本：不足以支撑有效风格分析，应要求补充或明确标记为低可信度
- 样本跨作者：先要求清洗样本，不要混合分析

### Input Validation Thresholds

- `valid sample`：同一作者、文本可读、长度达到 200 字以上
- `limited sample`：同一作者、100-199 字、可用于辅助但不能承载强结论
- `invalid sample`：少于 100 字、段落损坏严重、或无法确认作者归属
- 进行跨文本稳定性分析时，至少需要 2 个 `valid sample`；若不足，只能输出单篇或低可信度比较

## Analysis Workflow

### 1. 先做输入校验

- 判断是否为同一作者
- 判断样本数量是否足够
- 判断主题是否有变化，足够支撑“恒定 / 适应”分析
- 判断用户目标是偏分析、偏仿写，还是还要生成模板资产

如果用户没有明确主题分类，可先把样本按最贴近的写作场景归类，例如：评论类、汇报类、叙事类、反思类、动员类。若预设标签都不合适，可以自定义标签，但要在整个输出中保持一致。

### 2. 做单篇分析

每篇都同时分析宏观与微观层。

宏观层至少覆盖：

- 思维模式：作者如何切题、推进、收束
- 价值取向：立场、判断标准、关注重点
- 表达意图：想解释、说服、劝导、动员，还是记录
- 风格定位：理性、抒情、评论、纪实、议论混合等

微观层至少覆盖：

- 语言特征：句式、用词、修辞、标志性表达
- 结构特征：段落长度、转承方式、层次推进
- 叙事 / 论证特征：视角、论据组织、时序、例证方式
- 情感与节奏：语气浓淡、停顿、起伏、收束力度
- 个性标记：反复出现的表达习惯、意象、句法偏好

### 3. 做跨文本对比

把多篇文章放在一起，对每个维度做对比，不要只列相似点。

必须明确区分：

- 恒定特征：不管主题怎么变都反复出现的核心模式
- 适应特征：会随着主题、对象、场景变化而调整的部分

使用以下判定阈值，不要口头化地说“看起来挺稳定”：

- 当 `valid sample >= 3` 时，某特征至少出现在 **70% 及以上** 样本中，且不明显依赖单一主题，才可判为 `stable feature`
- 当只有 2 个 `valid sample` 时，某特征必须在 **两篇都出现**，且只能标为 `tentative stable feature`
- 某特征主要在特定主题、对象或场景中出现，或出现率在 **30%-69%** 之间，应判为 `adaptive feature`
- 出现率低于 **30%**、或仅见于单篇且无法确认原因的，放入 `unconfirmed finding`，不要强行归类

要特别观察：

- 哪些结构模式几乎每篇都出现
- 哪些词汇和句法是作者的长期偏好
- 哪些变化其实是主题驱动，而不是风格漂移
- 哪些冲突其实来自任务场景变化，而不是作者风格崩塌

### 4. 建立风格模型

把分析结果收束成一个可迁移的风格模型，至少回答：

- 作者写作时最稳定的思考方式是什么
- 作者最常用的组织框架是什么
- 作者如何在不同主题中保留个人特色
- 哪些表面特征可以模仿，哪些必须依附于思维方式才有意义

### 5. 根据任务输出结果

按用户需求组合以下输出：

- 风格分析结论
- 仿写指导
- 文体模板 / 风格画像

如果用户要求结构化输出或后续复用，优先使用 `@references/style-analysis-schema.md`。
如需给任何量表打分，先读取 `@references/scoring-anchors.md`。
在声称结果完成前，先对照 `@references/validation-checklist.md` 自检。

## Hard Rules

1. 没有足够样本时，不得把单篇观察包装成作者稳定风格。
2. 没有原文依据时，不得把判断写成确定事实。
3. 不得把低频特征、偶发修辞或一次性语句误判为恒定特征。
4. 不得把主题适应特征伪装成作者核心风格。
5. 当用户要求 JSON 时，不要为了“像真的一样”而删掉置信度、证据或 warning 机制。
6. 当证据不足时，应降低结论强度，而不是提高措辞强度。
7. 当样本之间存在明显冲突时，不得输出“统一作者风格总定义”“任何主题都照此写”之类的强结论；只能拆分为 `adaptive feature`、`unconfirmed finding` 或明确的多分支写法。
8. 当用户要求简化 JSON 或归档模板时，可以压缩字段数量，但不得删除证据边界；至少保留 `warnings`，且不得用占位内容冒充正式分析结果。

## Style Template Output

当用户希望后续复用、批量仿写，或明确提到 `office-docs-writer` / 模板 / 风格资产时，除了分析结论外，再额外产出一份结构化 style profile。

这份 style profile 不是“作者介绍”，而是可执行的写作约束集合。字段模板见 `@references/office-docs-style-template.md`。

生成这份模板时，至少保证：

1. 能说清适用场景，而不是笼统写“适合多数文章”
2. 能说清结构套路，而不是只写“逻辑清晰”
3. 能说清语气、节奏、句法、常用表达偏好
4. 能说清主题变化时哪些部分可调、哪些部分不能丢
5. 能说清禁区，避免 downstream 复用时走样

如果用户说“导入 office-docs-writer”，要明确说明：

- 你现在生成的是可适配的结构化风格资产
- 它可以在后续实现中映射为 guideline-like 或 template-like 资源
- 除非用户另外要求并提供明确集成任务，否则不要假装已经完成模块内的实际接线

## Output Contract

默认输出采用三层结构：

1. `风格判断`：先给作者风格核心定义、稳定性、适应性、独特性
2. `分析拆解`：给出恒定特征、适应特征、单篇观察与变化说明
3. `可执行结果`：给出仿写指南，必要时附 style profile

当用户明确要求 JSON、结构化存档、模板化复用时，按 `@references/style-analysis-schema.md` 输出。

结构化输出默认必须包含：

- `confidenceLevel`
- `evidenceQuotes`
- `sourceIndexes`
- `frequency`
- `warnings` 或 `unconfirmedFindings`

除非用户明确只要口语化摘要，否则不要删除这些字段。

如果用户要求“简化版 JSON”或“只保留结论”：

- 可以缩短说明文字，但不要把完整 contract 降级成无依据的空模板
- 至少保留 `warnings`
- 只要关键结论仍被保留，就仍需给出对应的 `confidenceLevel`、`evidenceQuotes`、`sourceIndexes`、`frequency`
- 如果用户拒绝保留这些证据边界，就改为自然语言摘要并明确说明“这不是正式风格归档结果”

当用户明确要求 office-doc 兼容模板时，再追加 `@references/office-docs-style-template.md` 对应的 style profile。

除非用户要求原样 JSON，不要生硬只贴 schema；应先给关键判断，再给结构化结果。

## Imitation Workflow

仿写时遵循以下顺序：

1. 先复现作者的思维切入方式
2. 再复现常用结构与段落推进
3. 再选择匹配的语气、句式、词汇偏好
4. 最后补上标志性表达，但不要堆砌表皮特征

给仿写指导时，要明确区分：

- 必须保留：思维模式、结构骨架、表达意图
- 可适配调整：主题相关表达、论据素材、场景措辞
- 不应机械复制：一次性语句、主题绑定词组、偶发修辞

### Expand Content In Author Style

“按作者习惯展开内容”不要停留在空话，至少按下面 5 步落地：

1. **复现切题方式**：先判断作者习惯是先亮观点、先铺场景、先抛问题，还是先给判断框架。
2. **映射主题路径**：把新主题映射到作者常用的论述路径，不要只替换关键词。
3. **搭结构骨架**：先列出开头、展开、转折、收束四段式或作者惯用变体，再写正文。
4. **按原有密度填充**：控制每段的信息密度、论据数量、句长节奏，保持和样本接近。
5. **最后做风格校准**：检查是否保留了作者的思维推进方式，而不是只保留词句表皮。

如果用户让你“写一篇像他一样的文章”，先给简短仿写策略，再输出正文；不要直接跳到成文而不解释依据。

## Failure Modes

- **样本不足**：明确提示只能输出单篇分析或暂定比较结论
- **样本冲突明显**：把冲突项移入 `adaptive feature` 或 `unconfirmed finding`，不要硬合并成唯一风格定义；不要给出“以后任何主题都按这一版统一写法执行”的总方案
- **主题过于单一**：提示适应性特征识别能力受限
- **OCR / 转写质量差**：先提示文本质量风险，再决定是否继续分析
- **用户要求去掉证据和置信度**：拒绝把无依据结论伪装成真实分析；可改为“简化版结构化摘要”或自然语言摘要，但必须保留 warning，且不得输出假装已填充完成的归档 JSON

## Common Mistakes

最常见错误有七类：

1. 把表层措辞当成风格本体，忽略作者的思维路径
2. 样本过少却下稳定风格结论
3. 不区分恒定特征与主题适应特征，导致仿写僵硬
4. 只会模仿“像”，不会模仿“为什么这么写”
5. 输出的 style profile 太空泛，无法被后续模块或人继续复用
6. 为了让 JSON 看起来完整，删掉证据、频次、置信度和 warning
7. 使用“稳定”“典型”“高频”这类词，但没有对应阈值和样本依据

发现这些问题时，不要轻描淡写。直接指出问题，再修正输出。

## Quick Checklist

输出前快速确认：

- 是否说明了样本数量与可信度边界
- 是否同时覆盖宏观层与微观层
- 是否明确区分恒定特征与适应特征
- 是否给关键判断附了原文依据、样本频次和置信度
- 是否把仿写重点放在思维方式和结构，而不是词句表皮
- 如果用户要模板资产，是否输出了结构化 style profile
- 如果提到 `office-docs-writer`，是否说明了“可适配”而不是伪称“已集成”
- 是否已对照 `@references/validation-checklist.md` 完成最终自检
